Rust 漫谈 1

本篇是 Rust 漫谈 的第一篇。这个系列主要记录一些学习 Rust 过程中的小想法和心得

还处于草稿阶段，具体的代码示例待补充

Rust 是如何借助所有权、借用、生命周期的概念优雅实现自动内存管理的？

在众多的编程语言中，内存管理（主要是垃圾收集）基本分成了两个流派：

一个是以 Java, Go 为代表的利用 GC 的语言。利用 GC 的一大好处是，程序员无需手动释放内存，减少了心智负担（？详见下文）。但是，在程序运行时，垃圾回收发生的时机是程序员无法预知的，在一些需要很高即时响应的领域，如金融交易领域，由于 GC 带来的"短短"几十毫秒（也许不准确）的停顿也是不可接受的（这个案例来源于 C++ 高级程序设计课上）。以 Java 来说，为了避免 "stop the world" 带来的糟糕体验，CMS、G1 收集器都在降低停顿时间上做出了一些进步，但是停顿是不可消除的。另外，GC 也不是万能的，在某些情况下，也会导致 out of memory 的情况发生，此时需要反复实验不同的虚拟机参数、垃圾收集器参数来解决问题，从而带来了更多的心智负担

另一个是以 C++ 为代表的手动内存管理的语言。这类语言将堆上内存管理的责任完全交给了程序员，对于新手而言，由于忘记 delete 而导致的内存溢出时有发生。但是对于老手而言，使用手动内存管理的语言的确是追求效率的最佳选择

关于以上两种流派的比较和争论从未停止。诚如周志明老师在《深入理解 Java 虚拟机》所说：

Java 与 C++ 之间有一堵由内存动态分布技术和垃圾收集技术所围成的高墙，墙外面的人想进去，墙里面的人却想出来

而独辟蹊径的 Rust 采用了与这两种流派均不同的内存管理方案。

  • Rust 采取了一种看似比较"极端"的内存管理策略：总是清除结束的作用域中所有的变量和对象，不管是在栈中还是堆中；除非在作用域结束时将想要继续使用的对象的所有权转移到作用域之外

  • 在 C++ 中，我们都被告知要向形参传递对象的引用（或者指针）而不是对象本身，因为 C++ 是按值传递，形参会拷贝实参，这会带来较大的内存开销，而且某些类型的对象是不可被拷贝的（如 ostream）；但是在 Rust 中，传递对象的引用而不是对象本身，是基于另一种考虑：

  • 如果函数的形参是一个对象类型而不是对象类型的引用，在传参时，实参的所有权被转移给了形参，如果函数结束时不将这个形参再传出来，将导致对象被 drop。这点与 C++ 不同，C++ 堆上的对象永远不会由编译器自动释放（智能指针是另一种技巧，这里不展开），除非显式地使用 delete
  • 使用借用 (borrow) 的概念解决上面这个问题：借用即引用，通过向形参传递对象的引用，形参并没有 own 这个对象本身，所以不会导致函数结束后对象就被drop的问题。同时，相对于 C++ 要手动添加 const 关键字来确保函数内部仅仅只是使用对象，而不是修改对象，Rust 中的所有引用创建时都是 immutable 的，除非显式地创建 mutable 的引用
  • 同样地，在 C++ 中，我们也被告知不要轻易返回函数中新创建的值或对象的指针或者引用，因为这会导致悬垂指针的问题，即想要通过指针访问到的对象已经被清理掉了。在 Rust 中，永远不会出现悬垂指针的问题，这是通过生命周期的概念保证的

  • 变量离开作用域后便自动调用 drop() 释放
