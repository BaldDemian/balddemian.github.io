Go

  • go test：在许多的编程语言中，可以通过注解的形式标记哪些函数是测试时自动执行的（比如 SpringBoot 中的 @SpringBootTest）。Go 中没有注解，于是用一套源文件和函数名的命名规则来标识测试函数

Task

修改 kvsrv 下的 client.go, server.go, commom.go

Implementation

  • Append 最初理解为是拉一个链表，但其实就是在原有的字符串后追加

  • RPC 标识符的处理

    最开始没有看到 client.go 中提供的 nrand() 函数，于是思考了一下标识符的实现方式。一个简单的实现是使用进程号 + 时间戳的方式。询问 GPT 后发现存在若干问题：

    1. 时间戳精度：如果时间戳的精度不够高（例如，只精确到秒），则在高频调用的场景下可能会发生冲突。
    2. 系统时间调整：如果系统时间被调整，时间戳可能会重复或回退，这可能导致 ID 冲突。
    3. 进程号重用：在长时间运行的系统中，进程号可能在进程结束后被操作系统重用，这可能导致进程号的重复。

    最终采取了 GPT 推荐的 UUID 的方式 nrand() 返回标识符

  • Go 的 map，访问一个不存在的键时，返回的是值类型的空值。比如本实验中，返回的 string 类型的空值是 ""。不过，某些情景下可能需要区分键存在，但值确实为 "" 和键不存在两种情况，这可以通过额外的一个布尔型的 map 实现

  • 处理不可靠的网络

    • 客户端的请求未到达。这种情况没有影响

    • 请求正确到达，但回复未到达，使得客户端继续重发请求。为了处理这种情形，服务端必须先缓存此次请求的返回值，如果后续收到同一请求，直接返回这个值，而不对 data 进行任何处理（否则就违反了一个请求只执行一次请求的规定）。同时服务端必须尽快释放客户端确认收到的回复所占用的内存，这就需要客户端的每次请求捎带最近一次已收到的回复的标识符

    • 同一请求多次到达

      这些请求的标识符是唯一的，因此可以识别出冗余的请求，进而返回事先缓存的结果，并且不对数据库进行任何修改

  • 如何释放服务端的内存？（TODO：找到更好的实现）

    在最开始的实现中，每个客户端的请求均捎带上一次请求的标识符，这样服务端在收到当前请求后，可以认为客户端已成功收到上一次的回复，进而删除对于上一次回复的缓存。但是，这会导致每个客户端的最后一次请求的回复的缓存是无法删除的！

    在一个用例中，涉及到一次 put 和许多次的 get，并且这些 get 是由不同的客户端一次性产生的（即这些客户端整个过程中只发送了这一次请求），这就使得上述的内存回收策略失效了！原因在于之前的实现中，对于 id 的管理是局限在 client.go 中的。如果转而在 common.go 中管理 id，那么不同客户端的请求可以捎带彼此的 ID。基于这样的实现，某些用例可以通过了，但还是存在两个无法通过的用例。百思不得其解，我明明删了那个重型对象了啊🤔。最后使用了一种十分丑陋的方法：每次成功收到回复之后，强制发送一个 ID 为非正常（即无法通过 getUUID() 函数生成，我选择了 "/a"）的 GET 请求，附上上次请求的 ID；在服务端，一旦识别到这个特殊的 ID，就能判断出这个 GET 请求并不是正常的请求，而是用来辅助释放内存的。这样的方式确实有些面向用例编程了

Result

Reflection

  • 对于内存该如何释放是一个值得深思的问题。一般的思路有超时机制、心跳检测（与某个客户端失联后，即删除与它的全部会话缓存）。针对本实验的某些用例，也许压缩存储重型对象也是可行的...
  • 如何在不可靠的网络上构建可靠的服务，是分布式系统需要解决一个重要问题。应当假设所有环节均可能出错，尽可能地处理不同 case，以构建一个鲁棒的系统或算法
